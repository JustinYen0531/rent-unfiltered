# Rent Unfiltered 透明租屋：專案整理

更新日期：2026-05-09

## 一、專案名稱

- 專案名稱：Rent Unfiltered
- 中文名稱：透明租屋

### 副標候選

- 看懂租約、費用與風險的 AI 租屋檢查工具
- 讓租屋不再靠運氣，讓風險變得看得見
- 不懂租約，也能看懂風險

## 二、專案定位

Rent Unfiltered 不是另一個租屋刊登平台，也不是主觀評論平台，而是一個面向租屋初心者的租屋資訊整合、風險判讀與報告生成系統。

本專案希望處理的不是「如何讓使用者看到更多物件」，而是「如何讓使用者真正看懂一個物件」。對學生、新鮮人與第一次租屋者而言，租屋風險往往不只來自價格，而是來自契約條款、費用揭露、居住安全、租客權益與資訊不完整。

因此，Rent Unfiltered 的核心目標是將原本分散、模糊、難以比較的租屋資訊，轉化為可閱讀、可比較、可追溯、可解釋的決策資訊，降低租屋資訊不對稱與搜尋成本。

### 定位定義

- 產品類型：租屋資訊透明化與風險判讀系統
- 主要對象：學生、新鮮人、第一次租屋者與資訊弱勢租客
- 核心任務：把分散的租屋資訊整理成可理解、可比較、可追溯的風險資訊
- 最終價值：讓租屋決策不只看價格與曝光，而能看見價格背後的風險

### 非目標

- 不提供租屋刊登與流量分發
- 不以主觀評論作為核心資料來源
- 不替使用者做最終決定
- 不提供法律保證、交易背書或安全保證

## 三、最終版本產品樣貌

最終版本的 Rent Unfiltered 會是一個以單一物件為中心的租屋風險分析系統。

使用者可以輸入或匯入一個租屋物件的資料來源，例如平台頁面、租約文字、房東提供資訊、公開資料或其他補充內容。系統會先將不同來源的資料整理成統一格式，再標示每個欄位的來源、揭露程度、衝突狀態與可信度，最後生成一份可供閱讀與比較的租屋風險報告。

這個產品的最終體驗，不是只給一個分數，而是讓使用者同時看到：

- 這個物件目前有哪些資訊已經清楚揭露
- 哪些資訊仍然未知或不完整
- 哪些資訊之間彼此矛盾
- 哪些條件可能構成契約、費用、安全或權益風險
- 哪些問題需要進一步詢問房東、仲介或自行查核

## 四、核心功能

### 1. 租屋資訊整合

- 接收來自不同來源的租屋資訊
- 整理平台欄位、租約條款、補充說明與公開資料
- 將分散資訊合併成同一份物件資料

### 2. RHIR 結構化標準化

- 將租屋資訊轉換為 RHIR，Rental Housing Interoperability Resources
- 以一致格式記錄房屋、契約、費用、設備、權益與環境資訊
- 保留每個欄位的來源、原文依據、揭露狀態與更新資訊

### 3. Transparency Layer 資訊透明層

- 記錄欄位是否已揭露、部分揭露、未揭露或無法判斷
- 記錄資訊來源是平台、房東、仲介、公開資料、系統推估或來源未知
- 記錄不同來源之間是否存在資訊衝突
- 區分「未揭露」與「不存在」，避免錯誤推論

### 4. 風險分析引擎

- 根據結構化資料計算 Rental Risk Index，RRI
- 依面向分析契約、費用、安全與權益風險
- 將風險分數與風險原因一起輸出，而不是只回傳單一結果

### 5. AI 輔助判讀

- 白話翻譯租約條款與複雜描述
- 標示可能需要特別注意的風險內容
- 協助產生追問房東或仲介的問題
- 生成使用者可直接閱讀的摘要與建議

### 6. 租屋風險報告

- 輸出單一物件的完整分析結果
- 提供可閱讀、可比較、可追溯的報告內容
- 支援後續保存、比較、匯出與討論

## 五、系統架構

Rent Unfiltered 的最終版本可分為五層：

### 1. Input Layer

- 使用者輸入資料
- 租約文字或檔案
- 平台物件資訊
- 外部公開資料

### 2. RHIR Normalization Layer

- 將不同來源資料轉成統一欄位格式
- 建立單一物件的 RHIR 記錄
- 保留欄位原始內容與標準化結果

### 3. Transparency Layer

- 記錄來源
- 記錄揭露狀態
- 記錄衝突
- 記錄可推估與不可判斷資訊

### 4. Risk Engine Layer

- 根據 RHIR 欄位進行風險分析
- 產生 RRI 分數、風險等級與分面向結果
- 將資訊不確定性與實際風險分開處理

### 5. Report and Explanation Layer

- 生成白話摘要
- 生成風險原因說明
- 生成簽約前檢查清單
- 生成追問建議與後續查核方向

## 六、資料架構

RHIR 是 Rent Unfiltered 的核心資料格式。它不只是記錄「房子有什麼」，而是記錄一筆租屋資訊如何被提供、如何被揭露、如何被判讀。

一份 RHIR 至少應包含以下結構：

- `property`：房屋基本資料
- `location`：位置與區域資訊
- `pricing`：租金與各項費用
- `contract`：契約條件
- `facilities`：設備與家具
- `safety`：安全相關資訊
- `rights`：租客權益與限制
- `provider`：資訊提供者身份
- `sources`：資料來源清單
- `transparency`：揭露、衝突、未知與推估狀態
- `riskScore`：風險分數與面向分數
- `aiFindings`：AI 分析結果
- `recommendations`：建議事項

每個重要欄位都應保留：

- 欄位值是什麼
- 這個值來自哪個來源
- 這個值的揭露程度如何
- 是否存在衝突或不確定性
- 是否需要後續人工確認

## 七、風險分析架構

RRI，Rental Risk Index，是 Rent Unfiltered 的核心分析模型，但它是整體系統中的一個模組，不是整個產品本身。

RRI 的目的不是替使用者做決定，而是提供一個可解釋的風險參考維度，讓租屋判斷不只停留在價格、地點或平台包裝，而能納入實際風險成本。

### 主要風險面向

- 契約透明度
- 費用透明度
- 居住安全
- 租客權益
- 生活適配度

### 契約透明度

- 是否有書面契約
- 是否有審閱期
- 是否明確約定修繕責任
- 是否明確約定提前解約條件
- 是否明確約定押金退還方式

### 費用透明度

- 租金
- 押金
- 水電費
- 管理費
- 網路費
- 清潔費
- 其他額外費用

### 居住安全

- 是否頂樓加蓋
- 是否違法隔間
- 逃生動線
- 消防設備
- 漏水狀況
- 電線安全
- 門鎖與出入安全

### 租客權益

- 是否禁止報稅
- 是否禁止遷戶籍
- 是否有稅負轉嫁
- 是否存在違法或不合理條款

### 生活適配度

- 通勤便利性
- 噪音
- 採光
- 垃圾處理
- 室友或鄰居條件
- 寵物限制

### 分數表示

- 0-20：低風險
- 21-40：中低風險
- 41-60：中風險
- 61-80：高風險
- 81-100：極高風險

### 分析原則

- 透明度分數與風險分數應分開處理
- 未揭露不等於不存在
- 生活適配度應避免與客觀風險完全混合
- 每個風險結果都應能回溯到資料來源與判斷依據

## 八、AI Agent 架構

AI 在 Rent Unfiltered 中不是最終決定者，而是資訊整理、風險揭露與白話轉譯的輔助系統。

### 1. 租約分析 Agent

- 分析租約中的風險條款
- 找出模糊、不完整或不利租客的描述
- 將條款翻譯成一般人可理解的白話內容

### 2. 費用透明度 Agent

- 分析租金與各種附加費用
- 標示費用是否清楚揭露
- 判斷是否可能存在不透明費用

### 3. 地區行情 Agent

- 比對區域租金行情
- 提供每坪租金或同類型物件的相對位置
- 協助辨識異常偏高或偏低的價格

### 4. 報告生成 Agent

- 整理風險重點
- 生成白話摘要
- 生成簽約前檢查項目
- 生成後續應追問與查核的建議

## 九、報告輸出

Rent Unfiltered 的最終輸出是一份單一物件報告，而不是只有分數頁面。

報告內容應包含：

- 總風險分數
- 風險等級
- 分面向分數
- 主要風險原因
- 白話解釋
- 資訊揭露狀態摘要
- 衝突欄位摘要
- 簽約前建議
- 待追問事項
- 待自行查核事項
- 條款逐條分析
- 租金行情比較

## 十、資料來源規劃

Rent Unfiltered 的資料來源可分為三類：

### 1. 主動揭露資料

- 房東提供資訊
- 仲介提供資訊
- 平台刊登欄位
- 使用者自行輸入資料

### 2. 公開資料

- 政府開放資料
- 內政部實價登錄租賃資料
- 內政部住宅租賃糾紛統計
- g0v 開放台灣民間租屋資料

### 3. 系統整合或推導資料

- 租約文字分析結果
- 租金行情比對結果
- 系統根據結構化資料產生的推估欄位

所有外部資料都應保留來源、授權、更新頻率與可追溯資訊，不應在未標示來源的情況下直接混入最終判斷。

## 十一、系統邊界與設計原則

### 系統邊界

- 不提供租屋刊登功能
- 不替使用者做最終租屋決定
- 不提供法律判決或安全保證
- 不以主觀評論作為核心資料來源

### 設計原則

- 透明的規則
- 可解釋的判斷
- 可被檢視的資料
- 可持續修正的風險模型
- 可保存、可交換、可比較的資訊格式

## 十二、最終目標

Rent Unfiltered 希望讓租屋不再只靠運氣、話術與曝光，而是讓資訊品質本身參與競爭。

它要解決的不是單純的搜尋問題，而是租屋資訊如何被量化、交換、解釋與信任的問題。當安全、契約透明、費用透明與租客權益能被更清楚地揭露與比較時，租屋市場就有機會從依賴經驗與包裝，走向依賴資訊品質與風險理解。

## 十三、可納入最終版的延伸參考

本節為獨立保留的延伸方向，不構成目前主架構的一部分，但未來可在不改變專案定位的前提下併入最終版本。

### 1. 更聚焦的早期目標使用場景

- 以台北校外租屋的大學生作為第一批驗證對象
- 以政大學生在文山區、木柵一帶找房作為早期測試場景
- 作為使用者訪談、資料蒐集與報告設計的優先場域

### 2. 更具體的資訊分散情境描述

- 補充目前常見來源，如 591、學校校外租屋網、Facebook 租屋社團、LINE 群組與仲介私訊
- 作為「資訊分散」問題的具體例證
- 用於簡報、企劃書與訪談問題設計

### 3. 可直接納入 RHIR 的重要欄位

- 是否可租補
- 是否能報稅
- 是否可養寵物
- 樓層與屋況
- 其他會明顯影響租屋判斷的條件欄位

### 4. 地區試點與驗證策略

- 先以單一學區或單一行政區作為試點
- 集中建立初步樣本、測試風險報告與驗證使用流程
- 作為最終版本擴展前的場域驗證方法

### 5. 未來商業化延伸方向

- 作為仲介、包租代管或租屋平台的風險揭露與報告生成工具
- 提供租約分析、資訊標準化與報告輸出能力
- 以分析層與標準層角色切入，而非直接轉型為租屋刊登平台

## 十四、目前已超出 MVP 的功能盤點

更新日期：2026-05-16

本節根據目前程式、文件與規格整理。這些功能已經超出 MVP 原本「單一物件表單輸入、轉成 RHIR JSON、產生基本 RRI 與白話報告」的最小範圍，可視為決賽版、Demo 強化版或後續產品化版本的功能候選。

### 1. 多頁面產品導覽與知識展示

- 已有首頁、專案導引、名詞解釋、RHIR 規格、資料庫與物件詳細頁等多個頁面，而不只是單一表單與結果頁。
- 名詞解釋與 RHIR 規格頁讓評審或使用者能直接理解 RHIR、RRI 與透明度設計。
- 專案導引頁讓作品不只展示工具，也展示問題意識、資料流程與專案定位。

### 2. RHIR JSON 預覽、複製、下載與欄位檢視

- 表單送出前可預覽 RHIR JSON。
- 詳細頁可依 RHIR 第一層節點查看 JSON 內容。
- 可下載 RHIR `.json` 檔案，讓 RHIR 成為可保存、可交換的資料格式。
- 可複製完整 RHIR JSON，方便測試、文件撰寫或外部分析。
- JSON 顯示具備語法高亮與 `disclosureStatus` 狀態標示，提升可讀性。

### 3. 版本管理與子版本流程

- 每筆租屋紀錄不只是一份單次輸入結果，而是可以有 `X`、`X-1`、`X-2` 等版本歷史。
- 詳細頁可切換不同版本，查看補件前後的完成度、RRI 狀態與說明。
- 可從既有紀錄新增子版本，並以既有 RHIR 欄位預填表單。
- 子版本不是覆蓋原紀錄，而是保存為新的版本節點，支援後續比較與追蹤。

### 4. 本機資料保存與匯入紀錄

- 使用者建立的 RHIR 紀錄會保存到瀏覽器本機儲存空間，不只存在於單次頁面狀態。
- 系統會把使用者新增的紀錄加入首頁列表，形成可返回查看的紀錄庫雛形。
- 已支援匯入 ID 與 URL hash 匯入流程，讓外部擷取資料可以進入表單。

### 5. 瀏覽器外掛擷取流程

- 專案已有 Chrome extension 雛形，可擷取目前頁面的可見文字。
- 外掛會產生匯入 ID，並可導向 Rent Unfiltered 表單載入擷取內容。
- 此功能讓資料來源從純手動表單擴展到 591、租屋平台、社團貼文或其他網頁文字。
- 外掛目前定位是使用者主動擷取，不是批量爬蟲，較符合資料倫理與展示安全。

### 6. 一鍵生成與一鍵補齊測試資料

- 表單提供一鍵隨機生成，可快速建立測試物件。
- 另有多組租約情境範本，可一鍵補齊空白欄位。
- 補齊流程只填入缺漏欄位，避免覆蓋使用者已填內容。
- 這讓 Demo 時能快速展示不同風險型態，不必每次手動輸入完整表單。

### 7. 待詢問欄位與現場確認問題

- 系統不只標示 `missing`，也會依欄位產生具體追問問題。
- 追問問題會標示影響面向，例如契約透明度、費用透明度、居住安全或租客權益。
- RHIR 預覽、欄位填寫情形與報告頁都能看到待詢問項目。
- 這已經從單純風險報告進一步變成看房前溝通清單與補件流程。

### 8. RRI v0.1 規則引擎

- RRI 已具備獨立規則引擎，不只是靜態展示分數。
- 引擎會依 RHIR 欄位即時計算總分、風險等級與分面向分數。
- `missing` 欄位會形成分數區間，區分已知風險與資訊不確定性。
- `conflict` 欄位會以較高風險處理，讓資料矛盾能直接反映在 RRI。
- 目前已納入契約透明度、費用透明度、居住安全、租客權益與生活適配度五個面向。

### 9. 生活適配度作為輔助風險面向

- MVP 原本不把生活適配度納入主分數，但目前 RRI 引擎已保留生活適配度面向。
- 生活適配度包含通勤便利、噪音、採光、垃圾處理、室友條件與寵物限制等項目。
- 此面向較接近個人需求與偏好，適合在決賽版中作為輔助分數或使用者客製化權重基礎。

### 10. Rule-based 結論生成層

- 除了 RRI 分數，系統已有固定格式的結論生成層。
- 結論會整理主要風險、資訊衝突、未揭露欄位與簽約前建議。
- 這一層不消耗 AI 算力，能穩定輸出可重現的白話說明。
- 它讓報告不只顯示數字，也能形成可閱讀的判斷文字。

### 11. AI Insight 風險脈絡解讀

- 報告頁已有 AI Insight 入口，可手動觸發。
- AI Insight 的定位不是重算分數，而是解釋風險組合、欄位關聯與優先追問順序。
- 已設計嚴格提示詞，要求 AI 不自行推測未揭露資訊、不做法律判定、不改變 RRI 分數。
- AI 輸出採結構化 JSON，包含風險摘要、風險模式、優先問題、新手白話與提醒語。

### 12. AI 租屋顧問對話

- 報告頁已有多輪 AI 顧問聊天功能。
- 使用者可針對單一物件詢問「最該擔心什麼」、「如何議價」、「怎麼跟家人解釋」等問題。
- AI 顧問會以 RRI 結構化結果作為回答依據，不重新計分。
- 這讓產品從單向報告延伸為可互動的租屋決策輔助工具。

### 13. API Key 與 AI 設定入口

- 介面已有設定視窗，可保存 OpenRouter API Key 到瀏覽器本機設定。
- 設定視窗會提示不要把真實 key 寫入前端原始碼。
- 這代表專案已開始走向可由使用者設定 AI 服務的模式，而不是單純靜態 Demo。

### 14. RHIR Server 與 Supabase 上傳

- RHIR 預覽視窗已提供上傳到 RHIR 資料庫的功能。
- Supabase 整合可將 RHIR JSON、物件標題、地區與租金寫入 `rhir_uploads`。
- 這讓 RHIR 從本機檔案進一步變成可上傳、可查詢的資料紀錄。
- 此功能已超出 MVP 的本機展示範圍，接近 RHIR Server / API 的早期雛形。

### 15. RHIR 資料庫管理頁

- 系統已有資料庫頁，可讀取所有已上傳的 RHIR 紀錄。
- 管理頁可查看上傳時間、Record ID、物件名稱、地區與租金。
- 可開啟單筆 RHIR JSON 並下載。
- 這讓系統開始具備後台管理與資料稽核能力。

### 16. 設計調整與 Demo 控制面板

- 專案已有 Tweaks Panel，可調整主色、密度與是否顯示 RHIR schema key。
- 這不是一般使用者必要功能，但對 Demo、視覺微調與評審展示很有幫助。
- 可在不改程式的情況下快速切換展示風格。

### 17. 文件與規格體系擴充

- 除 MVP 規格外，已整理 RHIR v0.1 人類可讀規範、JSON Schema、RHIR 整合 MVP 計畫、RRI 與分析報告實踐計畫等文件。
- RRI 文件已細到各欄位分數、AI 判斷提示、輸出格式與總分結構。
- RHIR 文件已開始區分欄位字典、資料來源、揭露狀態與未來 API / Server 發展路線。
- 這些文件讓專案從單一作品展示，擴展成一套資料標準與分析框架。

### 18. 最終版可優先整理成展示亮點的功能

- RHIR：從表單到標準 JSON，再到下載、上傳與資料庫查看。
- RRI：即時計算分數區間，清楚區分已知風險與未知風險。
- 版本管理：同一物件可補件、重建子版本並保留歷史。
- AI Insight：在規則分數之外提供白話脈絡解讀。
- 瀏覽器外掛：從外部租屋頁面擷取文字並匯入分析流程。
- 待詢問清單：把未揭露欄位轉成實際看房或簽約前可問的問題。
