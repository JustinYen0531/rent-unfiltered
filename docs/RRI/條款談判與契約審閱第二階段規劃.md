# XYZ 租屋生命週期與契約審閱規劃

> 目的：將 Rent Unfiltered 從「591 刊登資訊審查」延伸到看房、簽約、點交與入住後履約的持續風險決策支援。
>
> 定位：本文件是決賽展示與後續產品實作的第二階段規劃，不是法律意見或簽約保證。

## 一、為什麼需要第二階段

目前的 591 匯入與 RHIR 審查，主要處理找房階段：把刊登頁面的資訊整理成固定欄位，找出缺漏、矛盾與初步風險。

但押金、電費、修繕、提前解約、報稅與遷戶籍等案例，真正重要的問題通常發生在簽約前：房東如何回覆、雙方如何談條件，以及契約最後如何寫。第二階段因此不應只是增加更多風險分數，而應回答：

> 這個條件要問清楚什麼？哪些內容必須寫進契約？廣告、對話與契約是否一致？

## 二、XYZ 完整產品流程

```text
X  刊登判讀
  -> Y1 預約與看房準備
  -> Y2 現場看房
  -> Y3 條件談判
  -> Y4 契約與付款前
  -> Z1 點交與入住
  -> Z2 入住後履約、費用與修繕
```

第一版不處理退租與押金返還階段。使用者可以直接補登任何階段，系統只提示缺少的前置資訊，不阻擋真實世界中跳過或事後補登的情況。

X 建立「這間房目前公開了什麼資訊」；Y 驗證「現場看到什麼、對方承諾什麼、最後是否寫進契約」；Z 則持續記錄「點交後是否依約履行」。

## 三、第二階段 A：條款與費用談判

### 目標

在使用者簽約前，將 RRI 的風險欄位轉成可執行的談判與詢問清單。

### 優先欄位

| RHIR 欄位 | 常見問題 | 談判／確認方向 |
|---|---|---|
| `cost.deposit` | 押金金額與名目不清楚 | 確認是押金、定金或其他費用，以及返還條件 |
| `leaseTerms.depositRefundTerms` | 沒有返還期限或扣款規則 | 要求寫明返還期限、可扣款項目與計算方式 |
| `cost.electricityRate` | 每度單價、電表或公共電費不明 | 確認計價方式、電表型態與分攤規則 |
| `leaseTerms.repairResponsibility` | 房客被要求負擔全部維修 | 區分自然損耗、設備故障與人為損壞 |
| `leaseTerms.earlyTerminationClause` | 提前解約與違約金不明 | 要求寫明通知期限、違約金與可終止情況 |
| `rights.taxRegistrationAllowed` | 是否可報稅只用口頭說明 | 要求明確記錄可否報稅及相關費用負擔 |
| `rights.householdRegistrationAllowed` | 是否可遷戶籍未寫明 | 要求明確記錄可否設籍 |

### 輸出格式

每個高風險欄位至少產生：

- `riskSummary`：目前風險的白話摘要。
- `whatToAsk`：可以直接問房東或仲介的問題。
- `mustWrite`：建議寫入契約的條件。
- `redLine`：若對方拒絕釐清，應提高警覺的條件。
- `evidenceToKeep`：應保存的對話、截圖、照片或文件。
- `copyMessage`：可直接複製傳送的詢問話術。

系統應避免替使用者決定「一定要租」或「一定不能租」，而是把可接受條件、待確認事項與不可接受的紅線分開呈現。

## 四、第二階段 B：契約審閱與簽約

### 核心方法：三方比對

契約審閱不是只讀契約，而是比較三種來源：

```text
591 刊登內容
vs 房東／仲介對話與談判回覆
vs 契約條文
```

同一欄位若三個來源的內容不同，系統應標記為 `conflict`，並要求使用者在簽約前確認，不直接宣稱違法。

### 原文優先，欄位作為索引

第二階段不可只把整份契約分類成 RHIR / RRI 標籤後就丟棄原文。欄位與標籤的用途是搜尋、比較、計分與連結案例；契約原文才是審閱、引用與人工複核的主要依據。

每一個由文件抽出的欄位都至少應保存：

- `value`：標準化後的欄位值。
- `rhirField`：對應的 RHIR 欄位；尚無正式欄位時不得硬塞到相近欄位。
- `disclosureStatus`：`disclosed`、`partial`、`missing`、`conflict` 或 `unknown`。
- `sourceDocumentId`：原始契約或附件的識別碼與版本。
- `sourceText`：支持這個判讀的契約原句，不用摘要取代。
- `sourceLocator`：頁碼、條號、段落或字元範圍，讓使用者能回到原文核對。
- `confidence`：模型抽取信心；低信心或跨頁條款應進入人工確認。

原始檔案另存為不可被抽取結果覆蓋的版本；OCR、分段文字、欄位抽取與人工修正都屬衍生資料。報告呈現風險時，應同時顯示「欄位判讀」與「原文依據」，不能只顯示分類名稱。

### AI Insight 取得標籤與原文的資料流

AI Insight 不應自行掃描資料庫猜測要抓哪些標籤，也不應只接收目前的 RRI 摘要。建議由後端先完成可重現的查詢，再把查詢結果和原文片段一起交給 AI：

```text
契約原始檔
  -> OCR／版面與條款切分（保留頁碼、條號、位置）
  -> 欄位抽取候選（RHIR field + value + status + sourceText + locator）
  -> 規則驗證／必要時人工確認
  -> RRI rule engine 產生 riskType 與風險結果
  -> 以 rhirField + disclosureStatus + riskType 查 evidence_mappings
  -> 只取 review_status = verified 的 evidence_cases
  -> 組成 Insight Context（結構化標籤 + 契約原文片段 + 來源定位 + 案例）
  -> AI Insight 做脈絡解讀與 Action Plan，不改寫欄位或重算分數
```

也就是說，「抓標籤」的主鍵應是穩定的 `rhirField + disclosureStatus + riskType`，不是讓語言模型每次用模糊關鍵字重找。若使用者追問某條契約，再用 `sourceDocumentId`、條款位置與語意搜尋取回相關原文上下文；標籤負責縮小範圍，原文負責回答與舉證。

傳給 AI Insight 的 context 建議至少包含：

```json
{
  "document": {
    "id": "contract-001",
    "version": "draft-2",
    "rawTextAvailable": true
  },
  "findings": [
    {
      "rhirField": "leaseTerms.depositRefundTerms",
      "riskType": "deposit_dispute",
      "disclosureStatus": "conflict",
      "value": "退租後 60 天返還",
      "sourceText": "押租金於租賃關係消滅並完成點交後六十日內返還。",
      "sourceLocator": { "page": 7, "clause": "第十二條第三項" },
      "confidence": 0.96
    }
  ],
  "rriResult": {},
  "verifiedEvidence": []
}
```

受模型 context 長度限制，平常不必把整份合約全文塞入一次請求；但系統必須保存完整原文，並把每項判讀所依據的條款及前後文送入模型。遇到跨條款定義、附件或前後矛盾時，應擴大取回範圍，不能只讀單一句子。

### 比對結果

| 結果 | 意義 | 系統行動 |
|---|---|---|
| `disclosed` | 來源內容一致且已寫明 | 顯示已確認，保留來源 |
| `missing` | 契約或談判仍未提供內容 | 產生簽約前追問 |
| `partial` | 只有部分條件被寫明 | 指出尚未補齊的部分 |
| `conflict` | 廣告、對話與契約內容不一致 | 顯示衝突來源與待確認事項 |
| `unknown` | 文件無法判讀 | 提醒人工確認，不自行推論 |

### 優先檢查條款

- 租金、押金、定金與其他一次性費用。
- 電費、水費、管理費、網路費與公共費用分攤。
- 修繕責任、設備故障與自然損耗。
- 租期、續租、提前解約與違約金。
- 押金返還期限、扣款項目與點交方式。
- 報稅、租屋補助與遷戶籍權利。
- 房屋用途、轉租、寵物與訪客規則。

### 契約審閱輸出

契約審閱完成後，報告應分成：

1. **已確認條件**：廣告、對話與契約一致的內容。
2. **尚未寫明**：仍需要在簽約前補上的條款。
3. **前後衝突**：三方內容不一致的欄位與原始來源。
4. **簽約前 Action Plan**：修改條款、補充證據或暫緩簽約的建議。

## 五、RHIR 版本設計

### 欄位延續與擴充原則

第二階段應沿用第一階段已穩定的 RHIR 欄位，例如租金、押金、電費、修繕責任、提前解約、報稅與戶籍。這些欄位是跨來源比較的共同語言，才能直接比較 591 刊登、房東回覆與契約文字。

但契約不是另一張更長的 591 表單。第二階段還需要新增文件層資料，例如契約當事人、條號、定義、附件、簽署欄、條款間引用、原文位置與版本。這些資料應放在文件／條款層，再將其中可比較的內容映射回 RHIR；不要為了沿用舊架構而把所有契約內容強制塞入既有欄位。

RRI 則沿用既有風險面向與已驗證規則，但契約專屬的新判斷必須以新規則版本加入，並保留規則版本與依據。RHIR 負責描述資料，RRI 負責依規則判讀風險，兩者不應混成同一組「標籤」。

同一物件不覆蓋原始資料，而是保存不可變事件與每次事件後的累積快照：

```text
RHIR-物件-001
├─ X-01：591 刊登初始事件
├─ Y2-01：第一次現場看房
├─ Y3-01：房東談判回覆
├─ Y4-01：第一份契約草稿
├─ Z1-01：點交與入住
└─ Z2-01：第一次修繕事件
```

舊有 `X`、`X-1` 等版本視為 X 階段 legacy 事件，保留原顯示標籤。每個事件都應保留：

- 欄位值與 `disclosureStatus`。
- `sourceType` 與來源網址、檔案或對話紀錄。
- 建立時間與版本說明。
- RHIR delta、事件後累積 RHIR/RRI snapshot 與上一事件差異。
- 由哪個人提供或確認該資訊。
- `stage`、`substage`、`eventType` 與不可變的事件識別碼。

### 範例

```text
X-01：押金返還條件未揭露
Y3-01：房東表示退租後 30 天返還
Y4-01：契約寫退租後 60 天返還
結果：leaseTerms.depositRefundTerms = conflict
```

系統應顯示衝突的三個來源，並要求使用者在簽約前統一條款，而不是只更新成最新值。

## 六、案例庫在第二階段的角色

押金、電費、修繕與提前解約案例不是用來直接判定某位房東違法，而是用來說明「資訊不清楚時可能發生的後果」，並支援下一步行動。

資料流應為：

```text
RHIR 欄位 + disclosureStatus + riskType
  -> evidence_mappings
  -> verified evidence_cases
  -> risk explanation
  -> negotiation questions / contract checks / Action Plan
```

只有 `verified` 案例可以作為高可信度建議；`draft`、`revise` 或 `rejected` 案例不得直接進入使用者的高風險建議。

## 七、決賽展示流程

建議用同一間房完成以下展示：

1. 從 591 匯入一間物件。
2. 第一階段指出押金、電費或修繕欄位缺漏。
3. 顯示對應的真實案例與常見後果。
4. 產生談判前必問清單與可複製話術。
5. 匯入房東回覆或填入談判結果。
6. 匯入契約草稿或填入契約欄位。
7. 比較刊登、對話與契約的差異。
8. 顯示衝突欄位、應保留的證據與簽約前 Action Plan。

這條主線能清楚展示產品從「看懂租屋廣告」進一步變成「協助租客在簽約前驗證條件」。

## 八、實作順序

1. 完成 `evidence_cases` 與 `evidence_mappings` 的 Supabase schema 與 seed。
2. 將所有已確認案例建立正式 mapping，先聚焦押金、電費、修繕、提前解約與報稅／戶籍。
3. 在詳細頁加入 Related Cases 與來源追溯。
4. 建立談判用的 recommendation templates。
5. 加入 X／Y／Z 不可變事件、累積快照與跨來源欄位差異比較。
6. 將 AI 限制在 RRI、verified evidence 與 Action Plan 的說明層。
7. 用一個完整案例跑通「591 → 談判 → 契約 → 簽約前建議」的決賽 Demo。

## 九、邊界與風險

- RRI 與案例只能提供風險參考，不是法律認定。
- 系統不可把 `missing` 直接說成違法，也不可把單一案例當成普遍結論。
- 公開部署前，案例審核更新權限應改為 Supabase Auth + 受限 RLS；目前的無登入更新政策只適合內部原型。
- 契約原文與對話可能包含個人資料，必須使用私人登入、逐列 RLS、私有 Storage 與短效簽名網址。
