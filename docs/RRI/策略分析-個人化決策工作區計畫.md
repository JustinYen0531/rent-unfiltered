# 策略分析：個人化決策工作區計畫

> 狀態：Step 1–5 已完成；Step 6 改為承接 XYZ 生命週期
>
> 階段：原 Phase 4
>
> 定位：接在 RHIR、RRI 與 AI Insight 之後的個人化決策工作區
>
> 原則：不是分析報告的延伸，也不修改前面三個階段的客觀結果
>
> 最後更新：2026-07-18

---

## 一、為什麼要獨立

RHIR、RRI 與 AI Insight 主要描述「這個物件本身」：

```text
RHIR
-> 整理物件與契約資訊
-> RRI 計算客觀風險
-> AI Insight 解讀風險、案例與重要行動
```

策略分析描述的是「這位使用者要如何面對這個物件」：

```text
已完成的物件分析
+ 使用者自己的需求、限制與紅線
-> 個人化策略
```

兩者必須分開，原因如下：

* 同一個物件對不同租客可能產生不同決策。
* 使用者的預算、搬家急迫度與必要條件不應改寫 RHIR 或 RRI。
* AI Insight 應維持一般化、可分享、可回溯的物件解讀。
* 個人策略較私密，也可能隨時間與談判結果改變。

因此，策略分析應成為獨立區域，而不是繼續堆疊在分析報告下方。

---

## 二、產品流程與區域安排

完整流程規劃為：

```text
1. 欄位完成度
2. RHIR
3. 分析報告（RRI + Related Cases + AI Insight）
4. 策略分析
```

詳細頁已新增第四個主要頁籤：

```text
欄位完成度｜RHIR｜分析報告｜策略分析
```

### 分析報告的責任

分析報告停在以下內容：

* RRI 分數與風險區間
* 規則式結論
* Related Cases
* 三來源 AI Insight
* 可追溯的案例依據

### 策略分析的責任

策略分析才處理：

* 使用者自己的租屋情境
* 條件優先順序
* 可以談判與不能退讓的項目
* AI 顧問對話
* 個人化行動策略
* 房東拒絕或資訊不足時的備用方案

原本位於分析報告下方的「詢問 AI 顧問」已移入策略分析。

---

## 三、策略分析的輸入

策略分析不要求使用者重新填寫物件資訊，而是承接已經完成的資料。

### 系統既有輸入

* `recordId`
* `versionId`
* RHIR snapshot
* RRI 結果
* 已保存的 AI Insight
* AI Insight 使用的 verified cases
* 各項行動建議的來源模式

### 使用者情境

第一版只收真正會改變策略的少量資料：

* `rentalGoal`：租屋目的或使用情境
* `moveUrgency`：搬家急迫程度
* `budgetFlexibility`：預算彈性
* `mustHaveConditions`：必要條件
* `negotiableConditions`：可以妥協的條件
* `redLines`：不能接受的紅線
* `specialNeeds`：租補、寵物、短租、通勤或其他特殊需求
* `personalNote`：使用者想補充的自由文字

這些資料只影響策略排序與溝通方式，不回寫 RHIR、RRI、Risk Type 或案例資料。

## 四、策略分析的畫面結構

### 區塊 A：策略分析起點

畫面先顯示目前承接的分析狀態：

```text
RHIR：已完成
RRI：已計算
AI Insight：已保存 / 尚未保存
策略分析：尚未開始 / 進行中 / 已保存
```

如果尚未有可用的 RRI，應提示使用者先完成前一階段。AI Insight 尚未保存時，
策略仍可使用 RRI、verified cases 與個人情境產生，但會明確顯示 context 尚未包含
保存的 Insight。

### 區塊 B：我的情況

使用精簡表單收集個人條件。表單應：

* 優先使用選項與短文字
* 清楚說明這些資料不會改變物件的 RRI 分數
* 允許日後修改並重新產生策略

### 區塊 C：我的個人策略

使用者先填寫足夠的個人情境，再主動點選「生成個人策略」。右側會整理並自動保存：

* `decisionSummary`：目前最重要的決策摘要
* `priorityActions`：依使用者情境排序的行動
* `questionsToAsk`：簽約或看房前必問事項
* `negotiationPoints`：可以談判的條件
* `redLineWarnings`：觸及紅線時的提醒
* `evidenceChecklist`：應保存的文件與對話
* `fallbackPlan`：對方拒絕、無法確認或條件改變時的備用方案
* `copyMessages`：可複製但需依實際情況調整的溝通文字
* `strategyTrace`：這項策略使用了哪些個人條件、物件訊號、Insight 與案例

每次重新生成都建立新的 session，不覆蓋前一份結果。若使用者修改情境，舊策略會標示
需要重新生成，避免以新條件追問舊策略。

### 區塊 D：AI 策略顧問

只有在個人策略已生成並保存後，AI 策略顧問才開放追問。它使用：

```text
RHIR / RRI snapshot
+ 已保存 AI Insight snapshot
+ verified case references
+ 生成時的使用者情境 snapshot
+ 已保存的個人策略
+ 本次策略對話
```

AI 顧問可以拆解行動步驟、調整溝通語氣、解釋策略依據或討論備用方案，但不得在對話中
暗中改寫整份策略。若個人情境已修改，必須先重新生成策略才可繼續追問。

## 五、策略來源與排序原則

策略分析應保留 AI Insight 已建立的三種來源：

1. `mixed`：AI 對本物件的判斷與 verified 案例方向一致
2. `evidence_backed`：建議主要直接來自案例
3. `ai_assessment`：AI 根據 RHIR / RRI 作出的綜合判讀

策略分析可以依使用者情境重新排列「先做什麼」，但不能改寫來源類型。

例如：

* 租補是使用者紅線時，租補限制可以被提高策略優先級。
* 這不會讓它自動變成 `mixed` 或 `evidence_backed`。
* 案例支持程度與個人重要程度是兩個不同維度，畫面應分開呈現。

第一版建議增加個人策略優先級：

```text
立即處理
簽約前確認
可以談判
持續觀察
```

## 六、策略依據追蹤器

策略分析需要一個比 Evidence Tracker 更完整的 `Strategy Trace`。Evidence Tracker
回答「這項風險或建議用了哪些案例」，Strategy Trace 還要回答：

> 為什麼這項建議對這位使用者特別重要？

每一項個人行動策略都應能回溯以下鏈條：

```text
使用者條件或紅線
-> RHIR 物件事實或資訊缺口
-> RRI 風險訊號
-> AI Insight 行動與來源類型
-> verified case references（若有）
-> 個人策略優先級
-> 最終建議
```

### 畫面呈現

每張策略卡提供一個預設收合的入口：

```text
為什麼這樣建議？
```

展開後只顯示可核對的來源與簡短理由，例如：

```text
你的條件
租補資格是不能退讓的紅線

物件訊號
rights.rentalSubsidyAllowed 目前為 conflict

分析依據
RRI 將租客權益列為主要風險
AI Insight：mixed

案例參考
2 筆 verified cases 指向租補或報稅限制爭議

策略理由
因為這項條件同時是你的紅線，應在簽約前取得書面確認；
若房東拒絕，直接啟用備用方案。
```

這裡顯示的是來源追蹤與可讀的理由摘要，不保存或展示模型不可驗證的內部思考過程。

### 建議資料結構

```json
{
  "strategyTrace": {
    "personalInputs": [
      {
        "field": "redLines",
        "value": "必須可以申請租屋補助",
        "relevance": "直接影響是否承租"
      }
    ],
    "rhirSignals": [
      {
        "field": "rights.rentalSubsidyAllowed",
        "disclosureStatus": "conflict"
      }
    ],
    "rriSignals": ["租客權益為主要風險面向"],
    "aiInsightActions": [
      {
        "sourceMode": "mixed",
        "title": "簽約前取得租補書面確認"
      }
    ],
    "caseReferences": [
      {
        "caseId": "verified case id",
        "title": "verified case title"
      }
    ],
    "strategyPriority": "立即處理",
    "reasonSummary": "個人紅線與物件衝突欄位重疊，因此提高為簽約前必要行動。"
  }
}
```

### 追蹤規則

* 個人偏好、RRI 訊號、AI 判讀與案例支持必須分開標示。
* 沒有案例支持時，`caseReferences` 保持空陣列並明確說明。
* 案例 ID 與標題必須通過和 AI Insight 相同的 verified context 驗證。
* 策略優先級可以因個人條件提高，但不得因此改寫原本的 `sourceMode`。
* 每次保存策略 session 時，同時保存當次 trace，避免日後資料更新後無法解釋舊策略。

## 七、資料邊界與保存方式

### 個人情境草稿

使用者尚未生成策略前，個人情境會依 `recordId + versionId` 保存於瀏覽器
`localStorage`。這是單機草稿，不會跨裝置同步，也不會回寫 RHIR、RRI、Risk Type
或 AI Insight。

### Step 5 正式保存（已完成）

策略分析使用獨立資料表，不塞進 `ai_insights`：

```text
strategy_sessions
```

正式欄位：

```json
{
  "id": "uuid",
  "ownerToken": "uuid",
  "recordId": "string",
  "versionId": "string",
  "aiInsightId": "uuid",
  "promptVersion": "string",
  "strategyProfile": {},
  "rriSnapshot": {},
  "evidenceContext": {},
  "insightSnapshot": {},
  "strategyResult": {},
  "strategyTrace": [],
  "consultationMessages": [],
  "status": "draft | completed",
  "model": "string",
  "createdAt": "timestamp",
  "updatedAt": "timestamp"
}
```

保存原則：

* 一次策略分析是一個獨立 session。
* 同一物件與版本可以有多次策略 session。
* 每次生成會 INSERT 新 session，不覆蓋舊策略。
* 後續顧問對話只更新該 session 的 `consultationMessages` 與 `updatedAt`。
* 開啟策略分析時自動載入同一物件與版本的最新 session。
* 修改個人情境不覆蓋原本的 RHIR、RRI 或 AI Insight。
* 若 AI Insight 更新，舊策略仍保留原本引用的 `aiInsightId`。
* 導入正式登入前，可沿用目前瀏覽器 owner token；未來再轉為帳號所有權。
* RLS 的 SELECT、INSERT 與 UPDATE 都以 owner token 隔離。

## 八、第一版實作順序

Step 1–5 已完成；Step 6 保留為後續工作。

### Step 1：建立獨立入口（已完成）

* 新增「策略分析」頁籤或 route
* 顯示前置階段完成狀態
* 尚未完成前置資料時提供清楚提示

### Step 2：建立使用者情境表單（已完成）

* 先做最少必要欄位
* 不建立過度複雜的 persona 系統
* 驗證資料不會回寫 RHIR / RRI

### Step 3：移動 AI 顧問（已完成）

* 將分析報告下方的 AI 顧問移入策略分析
* 加入使用者情境與已保存 AI Insight
* 保留 verified case 引用限制

### Step 4：產生個人行動策略（已完成）

* 建立固定輸出 schema
* 顯示優先行動、談判點、紅線與備用方案
* 保留每項建議的來源與案例依據
* 為每一項策略建立可收合的 `strategyTrace`

### Step 5：保存策略 session（已完成）

* 建立 `strategy_sessions`
* 每次生成保存完成狀態與完整 snapshot
* 可以重新載入最新策略與後續顧問對話
* 個人情境改變後要求重新生成，不混用新舊 context

### Step 6：承接 XYZ 生命週期資料深化

Step 6 不再只是把多份策略並排比較，而是在使用者從刊登、看房、談判、契約到入住履約持續補充資料時，讓策略能清楚說明自己依據哪一個生命週期快照。

* 承接 `X / Y1 / Y2 / Y3 / Y4 / Z1 / Z2` 的最新累積 RHIR 與 RRI。
* 每份策略保存 `basisEventId` 與 `snapshotHash`，不因後續事件覆蓋。
* 新事件使原依據過期時，舊策略仍可翻閱，但顯示「資料已更新，需要重新生成」。
* 使用者主動生成新版策略；系統不因每次進度更新而自動花費 AI 成本。
* 歷史列表依生命週期事件排列，先回答「當時知道什麼」，再比較策略差異。
* AI 策略顧問只能追問目前已保存的策略 session；情境或生命週期資料改變後，必須先生成新版。

## 九、第一版不做的事情

為了避免 Phase 4 無限制膨脹，第一版明確不包含：

* 自動替使用者決定是否承租
* 法律判定或勝敗預測
* 自動聯絡或代表使用者與房東談判
* 向量 RAG 或大型相似案例推薦
* 複雜 persona 權重模型
* 自動修改 RHIR、RRI 或人工審核過的 Risk Type
* 把所有 AI 對話永久塞入分析報告

## 十、完成標準

第一版策略分析完成時，應符合：

* 策略分析是獨立區域，不增加分析報告的閱讀負擔。
* AI 顧問已移入策略分析。
* 使用者可以填寫少量個人條件。
* 系統能結合 RHIR、RRI、AI Insight、案例與個人條件。
* 產出可執行的個人策略，而不是另一份風險摘要。
* 個人偏好不會改寫客觀風險與案例來源。
* 每項重要策略都能展開「為什麼這樣建議」，並區分個人條件、物件訊號、AI 判讀與案例。
* 策略可以保存、重新載入並追溯使用的 AI Insight。

## 十一、開始實作的時機

Step 1–5 已完成。Step 6 隨 XYZ 事件與累積快照一起實作：

* X 為刊登判讀。
* Y1–Y4 涵蓋預約、看房、談判與契約付款前。
* Z1–Z2 涵蓋點交入住與履約修繕。
* 每次新增事件後保留舊 Insight 與策略，並以 basis event 判斷是否過期。
* 退租與押金返還不包含於第一版。

第一版先提供依事件回看與過期標示，不加入自動最佳化或自動替使用者決定是否承租。
