# RRI 與分析報告實踐計畫

> Rental Risk Index — 租屋風險指數  
> 版本：v0.1 草案  
> 狀態：進行中

---

## 一、總體設計原則

RRI 的目的不是替使用者做決定，而是提供一個**可解釋的風險參考維度**，讓租屋判斷能納入實際風險成本，而不只停留在價格、地點或平台包裝。

### 核心原則

- **透明度風險與實質風險分開處理**：資訊缺漏（missing）是不確定性風險，不等於真的有問題；但資訊衝突或明顯不利條款，才是實質風險。
- **未揭露不等於不存在**：missing 欄位不直接給最高風險，但因使用者無法判斷，應給予中高程度的不確定性加分。
- **conflict 比 missing 更嚴重**：資訊矛盾代表使用者可能被誤導，風險高於單純缺漏。
- **每個風險結果都能回溯來源**：分數必須能對應到具體欄位與判斷依據，不能只輸出一個數字。

### 分數方向

分數越高 = 風險越高。  
總分 100 分。

### 分數等級

| 分數 | 等級 |
|------|------|
| 0–20 | 低風險 |
| 21–40 | 中低風險 |
| 41–60 | 中風險 |
| 61–80 | 高風險 |
| 81–100 | 極高風險 |

## 三、共通欄位判斷規則

每個欄位都採用 5 級評分，不只分「有 / 無」：

| 狀態 | 加分 | 意義 |
|------|------|------|
| 明確揭露且內容合理 | 0 分 | 使用者能清楚理解條件 |
| 有揭露但不完整 | 1 分 | 有提到，但細節不足 |
| 模糊 / 需協商 / 依情況 | 2 分 | 無法在簽約前明確判斷 |
| 未揭露 / missing | 3 分 | 不代表不存在，但增加不確定性 |
| 衝突 / 明顯不利 / 高風險 | 4 分 | 資訊矛盾或對租客高度不利 |

## 四、契約透明度（20 分）

**依據**：內政部地政司住宅租賃契約書範本，包含租賃期間、租金、押金、修繕、提前終止、點交與返還等核心項目。

**配分**：5 個欄位，每欄最高 4 分，合計 20 分。

### 4.1 是否有書面契約 `hasWrittenContract`

**判斷邏輯**

| 情況 | 加分 |
|------|------|
| 明確有書面契約，且會簽正式租賃契約 | 0 |
| 有契約，但未說明是否為正式書面契約 | 1 |
| 僅說「可討論」「面議」「到時候再簽」 | 2 |
| 未揭露是否有契約 | 3 |
| 明確表示不簽書面契約、只口頭約定 | 4 |

**AI 判斷提示**

```
請判斷租屋資訊中是否明確提到會簽書面租賃契約。

判斷標準：
- 若明確寫「簽約」「書面契約」「正式租約」「使用租賃契約書」，判為低風險（0 分）。
- 若只寫「可簽約」但沒有說明契約形式，判為低到中風險（1 分）。
- 若寫「到時候談」「面議」「私下約定」，判為模糊（2 分）。
- 若完全沒有提到契約，判為 missing（3 分）。
- 若明確寫「不簽約」「不用契約」「口頭約定即可」，判為高風險（4 分）。
```

### 4.2 是否有審閱期 `reviewPeriod`

**判斷邏輯**

| 情況 | 加分 |
|------|------|
| 明確提供契約審閱期，或可先看契約再簽 | 0 |
| 有提供契約預覽，但未說明審閱時間 | 1 |
| 寫「現場看完可簽」「當天簽約」但未強迫 | 2 |
| 未揭露是否有審閱期 | 3 |
| 明確要求立即簽約、不得帶回看、不得審閱 | 4 |

**AI 判斷提示**

```
請判斷租屋資訊是否提供契約審閱或事前閱讀機會。

判斷標準：
- 若明確寫「可先看契約」「提供契約審閱」「可帶回審閱」「簽約前可確認契約內容」，判為低風險（0 分）。
- 若有提到契約，但沒有說明是否能事前審閱，判為部分揭露（1 分）。
- 若暗示看房當天可簽，但沒有強迫立即簽，判為中風險（2 分）。
- 若完全沒有提到審閱期，判為 missing（3 分）。
- 若明確要求「當場簽」「不能帶走契約」「不接受修改或審閱」，判為高風險（4 分）。
```

### 4.3 是否明確約定修繕責任 `repairResponsibility`

**判斷邏輯**

| 情況 | 加分 |
|------|------|
| 明確區分自然耗損由房東負責、人為損壞由租客負責 | 0 |
| 有提到修繕責任，但只寫大概原則 | 1 |
| 寫「依情況協商」「壞了再討論」 | 2 |
| 未揭露修繕責任 | 3 |
| 明確表示設備故障一律租客自費，或房東不負責修繕 | 4 |

**AI 判斷提示**

```
請判斷租屋資訊是否清楚說明修繕責任。

判斷標準：
- 若明確區分「自然耗損由房東負責、人為損壞由租客負責」，判為低風險（0 分）。
- 若只寫「房東負責基本修繕」「設備可報修」但沒有細節，判為部分揭露（1 分）。
- 若寫「依情況協商」「壞了再說」「雙方討論」，判為模糊（2 分）。
- 若完全沒有提到修繕責任，判為 missing（3 分）。
- 若明確寫「設備壞掉一律租客負責」「房東不負責修繕」「冷氣熱水器故障租客自費」，判為高風險（4 分）。
```

### 4.4 是否明確約定提前解約條件 `earlyTerminationClause`

**判斷邏輯**

| 情況 | 加分 |
|------|------|
| 明確寫提前通知期間、違約金或押金處理方式 | 0 |
| 有提到可提前解約，但細節不足 | 1 |
| 寫「提前解約另議」「依情況處理」 | 2 |
| 未揭露提前解約條件 | 3 |
| 明確規定提前解約押金全沒收，或條件明顯過重 | 4 |

**AI 判斷提示**

```
請判斷租屋資訊是否清楚說明提前解約條件。

判斷標準：
- 若明確寫「提前一個月通知」「違約金為一個月租金」「押金如何處理」等具體條件，判為低風險（0 分）。
- 若只寫「可提前解約」但沒有說明通知期間或費用，判為部分揭露（1 分）。
- 若寫「提前解約另議」「依房東規定」「到時候協調」，判為模糊（2 分）。
- 若完全沒有提到提前解約，判為 missing（3 分）。
- 若明確寫「提前解約押金全沒收」「未住滿不得退任何費用」且無合理說明，判為高風險（4 分）。
```

### 4.5 是否明確約定押金退還方式 `depositRefundTerms`

**判斷邏輯**

| 情況 | 加分 |
|------|------|
| 明確寫押金金額、退還時間、可扣款項目與標準 | 0 |
| 有寫押金金額，但退還條件不完整 | 1 |
| 寫「退租後檢查再退」「視情況扣款」但標準不明 | 2 |
| 未揭露押金退還方式 | 3 |
| 押金退還條件高度模糊、任意扣款、或明顯不退 | 4 |

**AI 判斷提示**

```
請判斷租屋資訊是否清楚說明押金退還方式。

判斷標準：
- 若明確寫押金金額、退租後多久退還、哪些項目可扣款、扣款標準，判為低風險（0 分）。
- 若只寫押金金額（例如「押二付一」），但沒有寫退還方式，判為部分揭露（1 分）。
- 若寫「退租後檢查再退」「視屋況扣款」「清潔費另扣」但沒有標準，判為模糊（2 分）。
- 若完全沒有提到押金退還，判為 missing（3 分）。
- 若明確寫「押金不退」「房東可自行決定扣款」「任何損耗均從押金扣除」且無標準，判為高風險（4 分）。
```

## 六、費用透明度（20 分）

**核心邏輯**：費用透明度的重點不是只知道月租，而是能估算**實際月成本**。水電費是最高風險項，因為最常「另計」卻不說清楚，入住後爭議也最多。

**配分**：7 個欄位，依重要性分配，合計 20 分。

| 欄位 | 滿分 |
|------|------|
| 租金 | 3 分 |
| 押金 | 3 分 |
| 水電費 | 5 分 |
| 管理費 | 3 分 |
| 網路費 | 2 分 |
| 清潔費 | 2 分 |
| 其他額外費用 | 2 分 |

### 共通判斷規則

| 狀態 | 加分比例 | 意義 |
|------|----------|------|
| 明確揭露且可計算 | 0% | 使用者可估算實際成本 |
| 有揭露但部分不完整 | 25% | 大致知道，但缺少細節 |
| 模糊 / 另計 / 面議 / 均分但無標準 | 50% | 無法準確估算 |
| 未揭露 / missing | 75% | 不代表有收，但增加不確定性 |
| 衝突 / 明顯不合理 / 高風險 | 100% | 資訊矛盾或可能造成重大費用風險 |

換算範例：3 分欄位 → 0 / 0.75 / 1.5 / 2.25 / 3；5 分欄位 → 0 / 1.25 / 2.5 / 3.75 / 5。  
內部可用小數，UI 顯示時四捨五入。

### 6.1 租金 `monthlyRent`（3 分）

| 情況 | 加分 |
|------|------|
| 明確寫月租金，例如 12000 元 / 月 | 0 |
| 有租金範圍，或依房型不同 | 0.75 |
| 寫「租金可議」「面議」「私訊報價」 | 1.5 |
| 未揭露租金 | 2.25 |
| 同一頁面租金前後不一致，或標題低價內文高價 | 3 |

**AI 判斷提示**

```
請判斷是否明確揭露月租金。

低風險：明確寫出固定月租金，例如「月租 12000」「租金 1.2 萬/月」。
部分揭露：有價格範圍，或依房型不同而有不同租金，仍能大致估算。
模糊：寫「面議」「可議」「私訊報價」「來電洽詢」而沒有具體金額。
missing：完全沒有租金資訊。
高風險 / conflict：同一物件出現互相矛盾的租金；標題或廣告寫低價，但實際條件顯示較高價格。
```

### 6.2 押金 `deposit`（3 分）

押金透明度只看**金額是否清楚**，押金退還條件已在契約透明度計分。

| 情況 | 加分 |
|------|------|
| 明確寫押金金額或押幾個月，例如押二付一 | 0 |
| 有寫押金，但未說明金額或月數完整細節 | 0.75 |
| 寫「押金面議」「依情況調整」 | 1.5 |
| 未揭露押金 | 2.25 |
| 押金資訊前後矛盾，或明顯高於常見範圍且無說明 | 3 |

**AI 判斷提示**

```
請判斷押金金額或押金月數是否明確。

低風險：明確寫「押二付一」「押金兩個月」「押金 24000」。
部分揭露：有提到押金，但未完整說明金額或計算方式。
模糊：寫「押金面議」「依情況調整」「看租期決定」。
missing：完全沒有押金資訊。
高風險 / conflict：不同位置押金說法矛盾；押金明顯超出常見範圍且沒有合理說明。
```

### 6.3 水電費 `electricityRate` + `waterFee`（5 分）

水電費是費用透明度最重要的一項。MVP 初版可將水費與電費合併為一個「水電費透明度」欄位。

| 情況 | 加分 |
|------|------|
| 明確寫電費與水費計算方式（台電計價、獨立電表、每度幾元、水費每月多少） | 0 |
| 有寫部分資訊（電費清楚但水費未明，或水費固定但電費另計） | 1.25 |
| 寫「水電另計」「依帳單」「均分」但沒有說明算法、戶數或每度價格 | 2.5 |
| 未揭露水電費 | 3.75 |
| 電費明顯偏高且無說明，或不同地方說法矛盾 | 5 |

**AI 判斷提示**

```
請判斷水費與電費的計算方式是否明確，使用者是否能估算每月水電成本。

低風險：明確寫「電費依台電帳單」「獨立電表」「電費每度 5 元」「水費每月 100 元」「水費依帳單均分且說明人數或戶數」。
部分揭露：電費有明確說明但水費沒有；或水費有固定金額但電費只寫另計；或寫含水費或含電費但另一項未說明。
模糊：寫「水電另計」「依帳單」「水電均分」「電費另計」但沒有每度價格、分攤方式或帳單依據。
missing：完全沒有水電費資訊。
高風險 / conflict：同一頁面同時寫「含水電」和「水電另計」；電費每度明顯偏高且無說明；寫法讓使用者無法判斷是否有獨立電表或是否被加價。
```

### 6.6 清潔費 `cleaningFee`（2 分）

## 七、居住安全（20 分）

**核心原則**：居住安全不是要 AI 判定違法，而是判斷「是否存在可辨識的安全疑慮、是否資訊不足、是否需要使用者進一步查證」。AI 不直接說「這一定違法」，而是說「此資訊顯示疑似風險，建議簽約前確認」。

**配分**：7 個欄位，依實體危害程度加權，合計 20 分。

| 欄位 | 滿分 |
|------|------|
| 是否頂樓加蓋 | 4 分 |
| 是否違法隔間 | 4 分 |
| 逃生動線 | 4 分 |
| 消防設備 | 3 分 |
| 漏水狀況 | 2 分 |
| 電線安全 | 2 分 |
| 門鎖與出入安全 | 1 分 |

### 共通判斷規則

| 狀態 | 加分比例 | 意義 |
|------|----------|------|
| 明確低風險 / 有安全說明 | 0% | 資訊清楚，無明顯疑慮 |
| 有揭露但不完整 | 25% | 有提到，但細節不足 |
| 疑似風險 / 模糊 | 50% | 可能有安全疑慮，需要確認 |
| 未揭露 / missing | 75% | 不代表有問題，但安全資訊不足 |
| 明確高風險 / conflict | 100% | 明確出現高風險描述或資訊矛盾 |

> 居住安全的 missing 比費用稍重，因為安全資訊若完全沒有，使用者很難在網路階段判斷。但仍然守住原則：**未揭露不等於危險，代表「安全不確定性」**。

## 八、租客權益（20 分）

**核心原則**：租客權益這組每個欄位都直接影響租客能否主張合法居住身份、申請補助與依法申報。AI 不直接做法律判定，而是標記「疑似限制」或「未揭露」。

**配分**：4 個欄位平均分配，各 5 分，合計 20 分。

| 欄位 | 滿分 |
|------|------|
| 是否禁止報稅 | 5 分 |
| 是否禁止遷戶籍 | 5 分 |
| 是否有稅負轉嫁 | 5 分 |
| 是否存在不合理條款 | 5 分 |

### 共通判斷規則

| 狀態 | 加分 | 意義 |
|------|------|------|
| 明確允許 / 無限制 | 0 分 | 權益沒有被限制 |
| 有條件允許 / 條件不完整 | 1 分 | 可接受，但需確認細節 |
| 模糊 / 需協商 / 未明確表態 | 2.5 分 | 權益狀態不明 |
| 未揭露 / missing | 3 分 | 不代表禁止，但增加不確定性 |
| 明確禁止 / 明顯不利 / 高風險 | 5 分 | 權益被限制或條款高度不利 |

> missing 給 3 分而非 5 分，因為未揭露不等於禁止。但這些屬於簽約前重要資訊，仍需提高不確定性風險。

### 8.3 是否有稅負轉嫁 `taxBurdenShift`（5 分）

> 稅負轉嫁不一定以「不能報稅」的形式出現，而是「可以報稅，但增加的稅金你負擔」，對租客是隱藏成本。

| 情況 | 加分 |
|------|------|
| 明確無稅負轉嫁，或可報稅且不另加費用 | 0 |
| 有提到報稅可能影響費用，但條件清楚且合理 | 1 |
| 寫「若有增加費用另議」「報稅費用另算」但不清楚 | 2.5 |
| 未揭露是否有稅負轉嫁 | 3 |
| 明確要求租客負擔房東因報稅、租補、設籍產生的稅金或額外成本 | 5 |

**AI 判斷提示**

```
請判斷租屋資訊是否存在稅負轉嫁條款。

稅負轉嫁指的是：房東要求租客承擔因報稅、租金申報、租屋補助、設籍等行為
所增加的稅金、費用或租金調整。

低風險：明確寫「可報稅且不加價」「申請租補不加租」「不另收費」。
低到中風險：有費用調整但條件明確（例如固定行政費或文件費），且不是轉嫁稅金本身。
模糊：寫「報稅增加費用另議」「稅金另算」「相關成本另計」但未說明算法。
missing：完全沒有提到稅負轉嫁。
高風險：明確寫「報稅增加的稅金由租客負擔」「申請租補需補貼房東稅金」
「設籍造成稅金增加由租客支付」「報稅需加租」。
```

### 8.4 是否存在不合理條款 `unfairTerms`（5 分）

> 此欄位在 UI 與文件中避免寫死「違法」，改用「疑似不合理或高度不利條款」。AI 不做法律判決，只標記疑似風險。

| 情況 | 加分 |
|------|------|
| 未發現明顯限制權益條款，或條款中性合理 | 0 |
| 有部分限制，但有清楚理由與範圍 | 1 |
| 條款模糊、保留過多單方決定權 | 2.5 |
| 未揭露相關條款內容 | 3 |
| 明確出現高度不利條款（任意進入、押金不退、設備全由租客負責、單方任意終止） | 5 |

**AI 判斷提示**

```
請判斷租屋資訊是否存在疑似不合理或高度限制租客權益的條款。
請注意：不要直接說「違法」，請使用「疑似不合理」「高度不利」「建議簽約前確認」等表述。

可能的高風險條款包括：
- 房東可不經同意進入租屋處
- 提前解約押金全數沒收且無合理條件
- 所有設備故障一律由租客負責
- 押金是否退還由房東單方決定
- 租客不得報稅、不得申請租補、不得設籍
- 房東可任意調漲租金
- 房東可任意終止租約
- 違約金或罰款明顯過高
- 限制租客基本居住使用但未提供合理說明

低風險：未發現明顯不合理條款，條件中性清楚。
低到中風險：有部分限制（例如禁菸、禁寵、不得轉租），但條件明確且屬常見管理規範。
模糊：條款使用「房東可視情況決定」「一切依房東認定」「租客不得異議」等單方用語。
missing：完全沒有揭露租約條款內容。
高風險：明確出現上述高度不利或疑似不合理條款。
```

## 九、生活適配度（20 分）

**核心原則**：生活適配度不用「風險」語氣，改用「適配疑慮」。這組較依個人需求，建議作為 RRI 的輔助維度，未來支援使用者偏好客製化權重。MVP 先使用一般學生預設權重。

**配分**：6 個欄位，合計 20 分。

| 欄位 | 滿分 |
|------|------|
| 通勤便利性 | 4 分 |
| 噪音 | 4 分 |
| 採光 | 3 分 |
| 垃圾處理 | 3 分 |
| 室友或鄰居條件 | 3 分 |
| 寵物限制 | 3 分 |

### 共通判斷規則

| 狀態 | 加分比例 | 意義 |
|------|----------|------|
| 明確適合 / 資訊清楚 | 0% | 條件清楚，對一般使用者友善 |
| 有揭露但不完整 | 25% | 有資料，但仍需確認 |
| 模糊 / 依個人需求 | 50% | 可能適合也可能不適合 |
| 未揭露 / missing | 75% | 不代表不好，但使用者無法判斷 |
| 明確不適合 / conflict | 100% | 與常見需求或使用者需求明顯衝突 |

> 生活適配度未來應支援使用者偏好權重。例如：有養寵物的使用者，寵物限制欄位權重應提高；對噪音敏感的使用者，噪音欄位權重應提高。MVP 階段先用固定權重。
