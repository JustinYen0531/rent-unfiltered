# 證據案例資料庫規格與管理頁規劃

## 目前流程位置

這是 RRI 證據層的第一階段：先建立可搜尋、可檢核、可被程式讀取的結構化案例庫。

本階段只處理兩件事：

1. Supabase 的 `evidence_cases` 與 `evidence_mappings` 資料表。
2. 軟體內的案例管理頁規格。

本階段暫不處理：即時爬蟲、embedding、向量資料庫、AI 自動判斷法律結論。

SQL 腳本位於：`docs/RRI/evidence_cases.sql`。

## 一、資料庫定位

案例資料不是直接塞在前端程式碼裡，而是放在 Supabase。軟體內的管理頁只是讓團隊用視覺化方式查看、篩選與審核資料。

這樣的分工是：

```text
Supabase
  ├─ evidence_cases       結構化案例卡
  └─ evidence_mappings    RHIR 欄位與案例的對應

Rent Unfiltered 管理頁
  ├─ 查看案例
  ├─ 依來源、年份、主題、審核狀態篩選
  ├─ 打開案例卡查看完整內容
  └─ 後續再加入新增、編輯與審核
```

前端只保存畫面與操作流程，不把案例 JSON 寫死在 `admin.jsx` 或其他 `.jsx` 檔案裡。這樣日後新增案例不需要重新部署前端。

## 二、第一版資料表

### `evidence_cases`

每一筆是一張「案例卡」，對應技能規格中的 canonical JSON。為了方便篩選，常用欄位會拆成資料庫欄位；同時保留完整 `case_record` JSON，避免未來欄位擴充時遺失原始結構。

重要欄位：

| 資料 | 用途 |
|---|---|
| `id` | 穩定案例 ID，例如 `research_scam_001` |
| `source_type` | `gov`、`court`、`research`、`social`、`interview` |
| `source_name`、`source_url` | 追溯原始來源 |
| `title`、`year` | 案例卡標題與年份 |
| `keywords` | 關鍵詞搜尋 |
| `rhir_fields` | 對應一至兩個 RHIR 欄位 |
| `risk_types` | 穩定風險類型，例如 `deposit_dispute` |
| `summary` | 一至三句案例摘要 |
| `common_outcome` | 可重複使用的常見後果 |
| `legal_basis` | 來源明確提到的法令或依據 |
| `action_hints` | 租客可以採取的行動 |
| `evidence_to_keep` | 建議保留的證據 |
| `confidence` | `high`、`medium`、`low` |
| `review_status` | `draft`、`verified`、`archived` |
| `notes` | 不確定處、整理備註 |
| `case_record` | 完整 canonical JSON 備份 |

### `evidence_mappings`

這張表負責回答：「當 RHIR 某欄位出現某個狀態時，要找哪些案例？」

例如：

```json
{
  "case_id": "research_scam_001",
  "rhir_field": "leaseTerms.depositRefundTerms",
  "disclosure_status": "missing",
  "risk_type": "deposit_dispute",
  "mapping_note": "押金或預約金條件未明確時，提示使用者先確認付款名目與退還條件。"
}
```

第一版先用一般索引與陣列欄位查找，不建立 embedding。這讓規則可以被檢查，也方便日後加入 RAG。

## 三、管理頁畫面規格

管理頁可以沿用目前 `admin.jsx` 的「資料庫」入口，但新增一個「證據案例」分頁或區塊，不修改現有 RHIR 上傳表格的用途。

### 1. 頂部摘要

顯示：

* 全部案例數
* `draft` 待審核數
* `verified` 已確認數
* 最近更新日期

### 2. 篩選列

第一版需要這四個篩選：

* 來源類型：政府、法院、研究、社群、訪談
* 主題：押金、電費、修繕、提前解約、報稅 / 戶籍 / 租補
* 年份：可輸入或選擇年份
* 審核狀態：草稿、已確認、封存

搜尋框先搜尋 `title`、`keywords`、`summary`，不先做語意搜尋。

### 3. 案例列表

列表每列顯示：

* 案例 ID
* 標題
* 來源類型與來源名稱
* 年份
* 對應 RHIR 欄位
* 可信度
* 審核狀態
* 「查看案例」按鈕

不要在列表直接塞完整摘要，避免畫面變成長篇文章；摘要放在詳細視窗。

### 4. 案例詳細視窗

點擊後顯示四個區塊：

1. 來源：標題、來源名稱、年份、原始連結。
2. 案例理解：摘要、常見後果、法律依據。
3. 可執行建議：`action_hints` 與 `evidence_to_keep`。
4. 系統連結：RHIR 欄位、風險類型、evidence mapping、可信度與審核備註。

原始文章或判決全文不放在列表中。管理頁只保存摘要與連結，讓來源可追溯但不把整篇內容複製進產品。

## 四、資料流

第一版建議採用這個順序：

```text
 helper / 團隊整理來源
        ↓
 canonical JSON
        ↓
 Supabase evidence_cases
        ↓
 管理頁檢查與標記 verified
        ↓
 evidence_mappings
        ↓
 RRI 分析結果出現欄位狀態時，顯示 Related Cases
```

管理頁第一版以「查看與篩選」為主。新增與編輯可以在確認登入方式後再開放，避免目前使用公開 anon key 時讓任何訪客修改核心證據。

## 五、團隊第一批資料清單

第一批先確認 30 筆，目的是驗證格式、來源追溯、RHIR 對應與案例卡閱讀流程，不代表台灣租屋糾紛的完整統計。官方混合型案例由程式自動同步，人工不再逐筆搜尋。

| 來源 | 目標 | 目前用途 | 還需要 |
|---|---:|---|---:|
| 崔媽媽基金會 | 6 | 看屋前付款、押金、修繕、契約與租客行動 | 6 減去已確認數 |
| 官方 GetG5 租屋候選 | 20 | 自動篩選押金、電費、修繕、提前終止與資訊揭露 | 20 減去已確認數 |
| 司法院裁判書查詢 | 4 | 押金扣款、電費或修繕責任 | 4 減去已確認數 |

目前第一筆是崔媽媽基金會的「看屋前付款」案例。官方 `GetG5` 資料是購屋與租屋混合案例，不是租屋專用資料庫；同步程式只把符合租屋語意的資料列為 `draft` 候選，經組員確認後才成為證據。來源入口與即時進度會顯示在軟體的「團隊證據工作台」。

第一批來源可以混合，但每筆仍要保留自己的 `source_type`、`source_url` 與 `confidence`。來源目標是工作配額，不是證據權威性的排名。

## 六、每筆案例 SOP

1. 確認來源網址、來源類型與發布年份。
2. 只整理一個主要爭議模式，不整篇複製文章或判決。
3. 填入一至兩個 RHIR 欄位與一個穩定 `riskType`。
4. 寫案例摘要、常見後果、租客行動與應保留證據。
5. 標記 `confidence`；不確定的法律判斷寫入 `notes`。
6. 第二位組員檢查來源、分類、欄位對應與措辭。
7. 通過後產生 seed SQL，寫入 Supabase 並標記 `verified`。

統計資料與案例資料要分開：統計回答「哪種糾紛常見」，案例回答「發生了什麼、租客可以保留什麼證據、下一步可以做什麼」。

## 七、完成條件

本階段完成的判準：

* Supabase 可以建立兩張表與必要索引。
* 一筆案例可以完整保存 canonical JSON 的必要欄位。
* 管理頁能依來源、主題、年份、狀態查看案例。
* 使用者能從案例卡回到原始來源。
* `draft` 與 `verified` 有清楚區別，未審核資料不直接作為高可信度建議。

## 八、後續銜接

完成這一階段後，下一步才是把 `evidence_mappings` 接到 RRI 結果，讓「欄位 + 狀態」可以查出 Related Cases。再之後才評估 embedding / RAG；不是每次使用者提問都重新上網搜尋。
