# RHIR 實作順序與資訊來源分類

更新日期：2026-05-10

## 一、核心觀察

租屋平台通常會優先呈現正面資訊，例如租金、照片、設備、交通便利、生活機能等。然而，對租客來說，真正困難的地方往往不是看見「有什麼」，而是判斷「什麼沒有被說出來」。

例如，若物件頁面沒有寫明對外窗、水電費算法、修繕責任或通風條件，不能直接推論它一定不存在，但也不能忽略這些資訊缺口。因為在租屋決策中，未揭露本身就是一種重要資訊。

因此，本專案不應直接將「未揭露」等同於「沒有」或「高風險」，而應建立一層資訊透明度判斷。

RHIR 常見欄位草案請參考：[RHIR常見欄位草案.md](RHIR常見欄位草案.md)。

## 二、資訊透明度分數

本專案可設計 Information Transparency Score，資訊透明度分數，用來衡量租屋資訊的揭露程度。

此分數不直接判斷物件好壞，而是判斷資訊是否完整、是否可驗證、是否足以支援使用者做決策。

### 判斷方式

不建議使用：

- 有對外窗：是 / 否

建議使用：

- 對外窗資訊：已揭露 / 部分揭露 / 未揭露

這樣可以避免把「未揭露」直接視為「不存在」，也避免對資訊不足的物件做出過度武斷的判斷。

### 初步狀態設計

- 已揭露：物件資訊中有明確說明。
- 部分揭露：物件資訊有提到，但內容不完整或不夠明確。
- 未揭露：目前資訊中沒有提供。
- 可由公開資料補足：房東未提供，但可透過政府或公開資料查詢。
- 可由系統推估：房東未提供，也未必有直接公開資料，但系統可透過周邊條件推導。
- 不可判斷：目前缺乏足夠資訊，不能下結論。

## 三、RHIR Transparency Layer

RHIR 應加入 Transparency Layer，用來記錄每個欄位的資訊來源與揭露狀態。

### 範例欄位

| 項目 | 狀態 | 說明 |
| --- | --- | --- |
| 電費 | 已揭露 | 平台或房東有提供計算方式 |
| 對外窗 | 未揭露 | 目前資訊中未提及 |
| 網路 | 未揭露 | 目前資訊中未提及 |
| 建材 | 未揭露 | 目前資訊中未提及 |
| 熱水系統 | 部分揭露 | 有提到設備，但未說明型式或費用 |

### 設計原則

- 不把「未揭露」直接等同於「沒有」。
- 不把「未揭露」直接等同於「高風險」。
- 將未揭露項目標示為「不確定性」。
- 讓 AI 說明哪些風險目前不可知。
- 讓使用者知道簽約前應該補問哪些問題。

## 四、租屋資訊來源分類

RHIR 收到的資訊可分為三類：房東或平台主動揭露、公共資料整合、系統主動搜尋或推導。

目前專案初期會優先處理第一類與第三類中的公共資料整合；系統主動搜尋與推導先記錄為未來功能，待作品成熟後再實作。

## 五、第一類：房東或平台主動揭露 Self-disclosed

這類資訊來自房東、房仲或租屋平台頁面，通常是使用者最先看到的資訊。

### 可能欄位

- 租金
- 押金
- 格局
- 坪數
- 樓層
- 租屋類型
- 家具
- 家電
- 是否可養寵物
- 是否可開伙
- 是否可報稅
- 是否可遷戶籍
- 水電費算法
- 管理費
- 網路費
- 最短租期
- 提前解約條款

### 注意事項

- 這類資訊容易受到廣告包裝與選擇性揭露影響。
- 有揭露不代表內容一定完整。
- 沒有揭露不代表一定不存在。
- 需要透過 Transparency Layer 標示揭露程度。

## 六、第二類：公共資料整合 Public data integrated

這類資訊不需要房東提供，而是透過政府資料、公開資料或可信資料源整合。

### 可優先考慮的方向

- 交通資訊
- 地區租金行情
- 建物基本資訊
- 災害潛勢
- 公共安全相關資料
- 周邊生活機能
- 法規或契約規範資料

### 可補足的內容

#### 交通透明度

- 最近捷運站距離
- 最近公車站距離
- 公車路線密度
- Ubike 站點距離
- 到學校或工作地點的通勤時間
- 夜間交通便利性
- 晚上 11 點後是否仍有可用大眾運輸

#### 地區租金行情

- 行政區租金中位數
- 每坪租金行情
- 同區域類似物件租金比較
- 是否明顯高於區域行情

#### 建物與法規資訊

- 屋齡
- 建物用途
- 是否疑似工業住宅
- 是否疑似頂樓加蓋
- 是否位於違建高風險區域
- 是否符合住宅用途相關規範

#### 災害與環境資訊

- 淹水潛勢
- 土壤液化潛勢
- 河岸或山坡地風險
- 火災或公共安全相關紀錄

#### 生活機能

- 超商距離
- 超市距離
- 醫院或診所距離
- 洗衣店距離
- 餐飲密度
- 學生生活所需設施距離

### 注意事項

- 公共資料需要確認授權與更新頻率。
- 公共資料不一定完整或即時。
- 需記錄資料來源，避免讓使用者誤以為是房東揭露。

## 七、第三類：系統主動搜尋或推導 System-inferred

這類資訊不是由房東提供，也不一定能直接從單一公開資料取得，而是由系統根據位置、周邊條件或其他資料推估。

此部分先列為未來功能，等作品成熟後再實作。

### 可能推導方向

#### 環境噪音

- 是否臨大馬路
- 是否靠近高架橋
- 是否位於商圈或夜市附近
- 是否靠近學校、工地或大型活動場所
- 周邊車流量推估

#### 居住安全

- 巷弄照明程度
- 是否位於死巷
- 附近派出所距離
- 逃生動線可疑程度
- 周邊避難空間
- 建築型態可能風險

#### 自然與居住環境

- 潮濕區域
- 日照方向
- 西曬可能性
- 通風條件推估
- 山區或河岸環境影響

#### 網路與通訊

- 5G 訊號覆蓋
- 光纖可用性
- 基地台距離
- 通訊品質推估

### 注意事項

- 系統推導應標示為推估，而不是事實。
- 推導結果需要可信資料支持。
- 若信心不足，應呈現為不確定性，而不是直接判定風險。

## 八、RHIR 初步資料架構方向

RHIR 應該能同時記錄三件事：

- 租屋條件本身。
- 欄位資訊來源。
- 欄位揭露與可信程度。

### 初步結構示意

```json
{
  "resourceType": "RentalHousingRecord",
  "version": "0.1",
  "property": {
    "rent": {
      "value": 18000,
      "source": "self_disclosed",
      "disclosureStatus": "disclosed"
    },
    "electricityFee": {
      "value": null,
      "source": "self_disclosed",
      "disclosureStatus": "missing"
    },
    "nearestTransit": {
      "value": "MRT station within 8 minutes walking",
      "source": "public_data_integrated",
      "disclosureStatus": "supplemented"
    },
    "noiseRisk": {
      "value": "unknown",
      "source": "system_inferred",
      "disclosureStatus": "not_evaluated"
    }
  },
  "transparency": {
    "knownFields": 2,
    "missingFields": 1,
    "supplementedFields": 1,
    "uncertainFields": 1
  }
}
```

## 九、建議實作順序

### 第一階段：確立 RHIR 格式

目標：

- 定義 RHIR 的基本欄位。
- 定義資訊來源類型。
- 定義揭露狀態。
- 定義透明度層。

重點：

- 先建立資料結構，不急著算風險分數。
- 先讓資料能被保存、交換、比較。

### 第二階段：引進公開資訊

目標：

- 找出可用的政府或公開資料。
- 補足房東不一定會提供的資訊。
- 優先處理交通、租金行情、建物、災害與生活機能等資料。

重點：

- 公開資料需要紀錄來源。
- 不將公開資料與房東揭露資訊混在一起。
- 明確標示資料更新時間與可信程度。

### 第三階段：建立初步 RHIR 樣本

目標：

- 建立一到數筆租屋資訊樣本。
- 將房東揭露資訊與公共資料整合進同一份 RHIR。
- 測試 Transparency Layer 是否能清楚呈現資訊缺口。

重點：

- 樣本不需要一開始很完整。
- 重點是能展示 RHIR 如何接收、補足與標示資料。

### 第四階段：補足主動搜尋與系統推導

目標：

- 加入交通便利性、環境噪音、生活機能、安全性、自然環境等推導欄位。
- 建立系統推估資訊的可信度標示。
- 將不可確定的資訊標成不確定，而不是直接判定。

重點：

- 此階段先列為後續功能。
- 等 RHIR 與公開資料整合穩定後再做。

### 第五階段：建立 Risk Index

目標：

- 在 RHIR 已能穩定記錄資訊後，再建立 Rental Risk Index。
- 風險分數應同時考量實際風險與資訊不確定性。
- 未揭露資訊不直接等於高風險，但可以提高不確定性提醒。

重點：

- Risk Index 應建立在資料結構之後。
- 透明度分數與風險分數應分開設計。
- AI 報告應說明「已知風險」與「未知風險」的差異。

## 十、目前優先順序總結

目前建議順序為：

1. 確立 RHIR 格式。
2. 引進公開資訊。
3. 建立初步 RHIR 樣本。
4. 補足主動搜尋與系統推導。
5. 建立 Risk Index。

此順序的原因是：若尚未建立清楚的資料格式與資訊來源分類，就直接進行風險評分，容易將「未揭露」誤判為「不存在」或「高風險」。因此，本專案應先處理資訊透明度，再處理風險判斷。
