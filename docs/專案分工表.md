# Rent Unfiltered 透明租屋：專案分工表

更新日期：2026-07-10

## 一、分工原則

本專案採取「產品技術落地」與「市場風險研究」雙主軸分工。

一方負責把想法做成可展示、可操作、可輸出的技術 Demo；另一方負責讓專案的問題定義、風險模型、資料依據與市場意義更加完整。

這份分工不是固定死的工作清單，而是目前討論階段的參考分配。後續可以依照比賽需求、開發進度與組員擅長領域調整。

## 二、我：產品 / 技術主導

### 角色定位

負責讓 Rent Unfiltered 從概念變成可以實際操作的技術 Demo。

主要任務是把租屋風險檢查、AI 分析、報告生成、資料格式與使用者介面落地，讓作品可以被展示、被操作、被理解。

### 主要負責內容

- AI Agent
- 前端介面
- JSON schema
- 系統架構
- 租約分析功能
- 報告生成功能
- Demo 流程
- UI 設計
- Pitch 視覺化

### 具體工作方向

- 建立使用者輸入租屋條件的表單。
- 設計租屋風險分數的前端呈現方式。
- 將風險結果轉成白話提醒。
- 製作租屋檢查清單與建議區塊。
- 規劃 AI Agent 如何分析租約文字。
- 設計租屋資訊 JSON 或 RentRisk JSON 的資料結構。
- 建立可展示的 Demo 流程。
- 將組員整理出的風險規則與資料依據整合進系統。
- 製作簡報中需要的產品畫面、流程圖與展示素材。

### 核心責任

- 讓專案做得出來。
- 讓 Demo 能順利展示。
- 讓使用者可以看懂操作流程。
- 讓 AI 分析結果可以被視覺化與報告化。

## 三、組員：場域驗證 / 外部合作 / 專家研討主導

### 角色定位

負責讓 Rent Unfiltered 不只停留在內部想像，而是真正接到租屋、地政、學術與實務場域。

他的核心價值不是寫程式，也不是單純幫忙查資料，而是作為一個能接觸較高學術背景與外部實務人士的人脈型角色，替專案打開更可信的討論、檢核與驗證入口。

主要任務是把專案帶進真實對話與真實驗證流程，讓 RHIR、RRI、Evidence 與 Action Plan 的設計，不只來自團隊推論，也能經過專家與使用場域的修正。

### 四個主要角色

#### 1. 場域驗證與外部合作負責人

負責替專案建立第一層真實接點，讓 Rent Unfiltered 可以接觸到租屋與地政相關場域，而不是只在內部閉門設計。

**主要職責**

- 接觸地政系老師、不動產或租賃相關課程教師、地政事務所或地政局相關人員。
- 接觸租屋服務、包租代管、學校住宿輔導單位、學生組織或租屋服務相關窗口。
- 協助安排專案簡介、訪談、討論或非正式請益機會。
- 幫助團隊理解哪些單位適合先交流、哪些合作方式較現實、哪些說法較容易被接受。

#### 2. 專家訪談與學術研討協調人

負責把專案帶進更高學術背景或更有實務經驗的人面前，讓我們的想法能被真正懂租屋制度、土地管理、住宅市場與租賃實務的人檢視。

**主要職責**

- 邀請 3 到 5 位具有相關背景的人，對 RHIR 欄位、RRI 規則與 Evidence 架構進行討論或檢核。
- 整理訪談提綱，聚焦在「哪些欄位最重要」「哪些風險最常被忽略」「哪些條件最需要留證」。
- 記錄專家對風險分數、權重與判斷邏輯的修正意見。
- 協助把這些外部意見轉成可引用的設計依據、簡報論述或文件更新方向。

#### 3. 案例蒐集與風險依據整理負責人

負責把外部接觸到的經驗、案例與研究，轉成真正能支撐產品的風險依據，而不是停在零散的聊天印象。

**主要職責**

- 蒐集租屋糾紛、押金爭議、電費爭議、漏水修繕、提前解約、報稅與戶籍限制等真實案例。
- 整理哪些爭議最常發生、最常在哪個階段爆發、租客最常缺什麼證據。
- 協助對照政府資料、專家說法、社群痛點與學生經驗，整理成 Evidence Layer 的基礎素材。
- 把案例整理成團隊可用的摘要，而不是只有原始連結或散亂筆記。

#### 4. 使用者招募與試辦推進負責人

負責讓專案真正進到目標使用者手上，驗證它到底有沒有幫助，而不是只在團隊內覺得方向正確。

**主要職責**

- 招募正在找房的學生、已有租屋經驗的學生、曾遇過租屋爭議的人作為第一批測試者。
- 協助安排小規模試用、訪談、回饋收集與後續追蹤。
- 推動真實房源試辦流程，例如讓使用者以實際正在看的房源建立 RHIR、補件、建立子版本。
- 整理使用者在哪些地方更能辨識風險、哪些追問最有幫助、哪些地方仍然卡住。

### 主要負責內容

- 外部合作與關係建立
- 專家訪談與檢核安排
- 場域驗證與試辦推進
- 使用者招募與回饋整理
- 風險案例與證據依據蒐集
- 學術與實務討論紀錄整理
- Evidence Layer 的素材支持
- 簡報中「可信度」「需求驗證」「外部回饋」相關論述

### 核心責任

- 讓專案接上真實場域。
- 讓 RRI 與 Evidence 設計有外部依據。
- 讓作品能被更高學術背景與實務人士檢視。
- 讓產品不只是技術 Demo，而是開始具備可驗證、可合作、可研討的基礎。

## 四、共同討論項目

以下內容需要兩人一起討論，不能完全由單方決定。

- Rental Risk Index 的最終面向。
- 每個風險面向的權重。
- 哪些條款或租屋條件算高風險。
- 風險分數如何分級。
- 是否納入生活適配度。
- 是否使用政府開放資料。
- 是否加入地區租金行情比較。
- 是否設計 RentRisk JSON 作為專案亮點。
- MVP 最終功能範圍。
- 比賽 Demo 的展示流程。
- 簡報主軸與故事線。

## 五、合作方式

### 我提供

- 可操作的產品原型。
- AI 分析與報告生成能力。
- 技術實作與 Demo 整合。
- 視覺化呈現與簡報素材。

### 組員提供

- 場域接觸與外部合作入口。
- 專家檢核與學術研討機會。
- 真實案例與證據依據整理。
- 使用者招募與試辦推進。
- 專案可信度與需求驗證支持。

### 最終共同產出

- 一個能操作的 Rent Unfiltered 技術 Demo。
- 一套可解釋的租屋風險指數。
- 一份白話化租屋風險報告。
- 一個完整的比賽簡報故事。
- 一個能說明社會價值與市場需求的專案。

## 六、一句話總結

我負責讓 Rent Unfiltered 做得出來，組員負責讓 Rent Unfiltered 接上真實場域、專家與使用者，最後一起把它變成一個真正站得住、能被信任的作品。
