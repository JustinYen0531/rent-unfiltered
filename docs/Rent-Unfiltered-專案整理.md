# Rent Unfiltered 透明租屋：專案整理

更新日期：2026-05-09

## 一、專案名稱

### 一定會做

- 專案名稱：Rent Unfiltered
- 中文名稱：透明租屋

### 可能使用的副標

- 看懂租約、費用與風險的 AI 租屋檢查工具
- 讓租屋不再靠運氣，讓風險變得看得見
- 不懂租約，也能看懂風險

## 二、核心定位

### 一定會做

- 不是做「房地產專家工具」，而是做「幫普通人理解租屋風險」的工具。
- 目標使用者是學生、新鮮人、第一次租屋的人。
- 把複雜的租約、費用、權益、居住條件轉成白話風險提醒。

### 可能會做

- 延伸成像 FHIR 一樣的租屋資訊交換格式。

## 三、主要功能方向

### 一定會做

- 青年租屋風險檢查器。
- 使用者輸入或勾選租屋條件。
- 產生租屋風險分數。
- 產生風險提醒。
- 產生簽約前檢查清單。
- 產生白話建議。

### 可能會做

- AI Agent 分析租約。
- 上傳 PDF、圖片或貼上租約文字。
- 生成完整租屋風險報告。
- 比對地區租金行情。
- 爬蟲或使用外部租屋資料。

## 四、風險指標設計

### 一定會做

- 建立自己的租屋風險指數。
- 名稱可能是：RRI：Rental Risk Index。
- 分數設計：0 到 100 分，分數越高代表風險越高。

## 五、RRI 可能包含的五個面向

### 一定會做：契約透明度

- 是否有書面契約
- 審閱期
- 修繕責任
- 提前解約
- 押金退還方式

### 一定會做：費用透明度

- 租金
- 押金
- 水電費
- 管理費
- 網路費
- 清潔費

### 一定會做：居住安全

- 是否頂樓加蓋
- 是否違法隔間
- 逃生動線
- 消防
- 漏水
- 電線
- 門鎖

### 一定會做：權益風險

- 是否禁止報稅
- 是否禁止遷戶籍
- 是否有稅負轉嫁
- 是否有違法或不合理條款

### 可能會做：生活適配度

- 通勤
- 噪音
- 採光
- 垃圾處理
- 室友或鄰居
- 寵物限制

## 六、風險等級

### 一定會做

- 0-20：低風險
- 21-40：中低風險
- 41-60：中風險
- 61-80：高風險
- 81-100：極高風險

## 七、可能的資料來源

### 一定會做

- 使用者自行填寫資料。

### 可能會做

- 租約文字分析。
- 政府開放資料。
- 內政部實價登錄租賃資料。
- 內政部住宅租賃糾紛統計。
- g0v 開放台灣民間租屋資料。
- 591、樂屋網、好房網等租屋網站資料。

### 暫時不一定會做

- 大量爬商業租屋網站。

## 八、AI Agent 架構

### 可能會做：租約分析 Agent

- 分析風險條款。
- 找出模糊條款。
- 白話翻譯租約內容。
- 產生詢問房東的建議。

### 可能會做：費用透明度 Agent

- 分析租金、水電、管理費、清潔費、網路費。
- 判斷是否有額外費用不透明。

### 可能會做：地區行情 Agent

- 比對地區租金行情。
- 判斷每坪租金是否偏高。

### 可能會做：報告生成 Agent

- 產生完整租屋風險摘要報告。

## 九、報告內容

### 一定會做

- 總風險分數。
- 風險等級。
- 主要風險原因。
- 白話說明。
- 簽約前建議。

### 可能會做

- 分面向分數。
- 條款逐條分析。
- 租金行情比較。
- 可匯出的報告。

## 十、租屋資訊 JSON 交換格式

### 可能會做

- 設計類似 FHIR 精神的租屋資料交換格式。

### 可能名稱

- RHIR：Rental Housing Interoperability Resources
- RentRisk JSON
- 透明租屋交換格式

### 可能內容

- property：房屋基本資料
- contract：契約條件
- riskScore：風險分數
- aiFindings：AI 發現的問題
- recommendations：建議事項

### 可能輸出格式

- .json
- .rent.json

## 十一、目前最小可行版本 MVP

### 一定會做

- 表單輸入。
- 租屋風險分數。
- 三到五個風險提醒。
- 租屋檢查清單。
- 白話解釋區。

### 可能會做

- 租約上傳。
- AI 條款分析。
- JSON 匯出。
- 風險報告下載。
- 地區租金行情比較。

## 十二、比賽簡報可用的核心說法

### 一定會用的方向

- 本作品從學生與年輕人的租屋問題切入。
- 目標是降低租屋資訊不對稱。
- 將租約、費用與風險轉化為一般人看得懂的分數與建議。

### 可能會用的方向

- 延續 FHIR 專案的 AI Agent 分析架構。
- 將複雜資訊轉成可交換、可分析、可生成報告的結構化資料。

## 十三、不建議直接交給 AI 決定，需組員自行查證與判斷的內容

以下項目屬於「有討論空間、需要資料來源、需要組員判斷」的部分。日後查到資料時，可以補在各項目的「待補資料」欄位。

### 1. RRI 權重與計分規則

目前狀態：需要組員自行判斷。

原因：

- 每個風險面向的權重會影響最後分數。
- 權重需要根據法規、糾紛統計、使用者訪談或專家意見調整。
- AI 可以協助整理，但不應直接決定分數是否合理。

待補資料：

- 租屋糾紛常見類型資料
- 法規依據
- 使用者訪談結果
- 組員討論後的權重版本

### 2. 哪些條款算高風險

目前狀態：需要組員自行查證。

原因：

- 「禁止報稅」、「禁止遷戶籍」、「押金超過合理範圍」、「提前解約沒收押金」等項目，需要確認實際法規或契約規範。
- 不同條款可能不是單純違法，而是需要看情境。
- AI 的判斷可能不夠精準，需要資料來源支持。

待補資料：

- 住宅租賃定型化契約應記載及不得記載事項
- 租賃相關法規
- 消保案例或法律文章
- 組員整理的條款風險表

### 3. 風險等級區間是否合理

目前狀態：需要討論。

原因：

- 0-20、21-40、41-60、61-80、81-100 目前只是初步分級。
- 實際使用時可能需要根據測試案例調整。
- 高風險與極高風險的界線需要有可說明的理由。

待補資料：

- 測試案例
- 不同分數結果的人工檢查
- 組員對分級文字的討論紀錄

### 4. 生活適配度是否納入 RRI

目前狀態：可能會做，需討論。

原因：

- 通勤、噪音、採光、垃圾處理、室友或寵物限制，偏向生活偏好，不一定是法律或契約風險。
- 若納入總分，可能會讓風險分數混合「客觀風險」與「個人偏好」。
- 需要決定它是總分的一部分，還是獨立顯示。

待補資料：

- 使用者需求訪談
- 組員對「風險」與「適配度」的定義
- 是否分成 Rental Risk Score 與 Living Fit Score

### 5. 政府開放資料是否適合使用

目前狀態：可能會做，需查證。

原因：

- 需要確認資料是否開放、格式是否能讀取、更新頻率是否足夠。
- 需要確認資料欄位是否真的能支援租金行情或風險判斷。
- 不能只因為資料存在，就假設它適合本專案。

待補資料：

- 資料來源網址
- 授權條款
- 欄位說明
- 更新頻率
- 初步資料樣本

### 6. 是否爬商業租屋網站

目前狀態：暫時不一定會做，需謹慎判斷。

原因：

- 商業網站可能有使用條款、反爬蟲機制或資料授權限制。
- 技術成本可能高於 MVP 需求。
- 若要放進比賽作品，需要避免造成法律與倫理風險。

待補資料：

- 各網站使用條款
- 是否允許自動化存取
- 是否只做少量測試
- 是否改用政府資料或開放資料替代

### 7. g0v 開放台灣民間租屋資料是否可用

目前狀態：可能會做，需查證。

原因：

- 需要確認資料是否仍可取得。
- 需要確認授權方式、資料品質與欄位完整度。
- 需要判斷是否適合本專案的風險分析。

待補資料：

- GitHub 或資料集網址
- 授權方式
- 欄位內容
- 資料更新狀況
- 是否能對應到本專案的分析需求

### 8. AI Agent 要做到什麼程度

目前狀態：可能會做，需討論。

原因：

- 租約分析 Agent、費用透明度 Agent、地區行情 Agent、報告生成 Agent 都是可行方向，但不一定要全部做。
- 需要依照時間、人力與比賽展示需求決定優先順序。
- 若 Agent 太多，可能增加系統複雜度。

待補資料：

- 組員分工
- 開發時間
- MVP 必要功能
- Demo 展示流程

### 9. 租屋資訊 JSON 交換格式是否納入主軸

目前狀態：可能會做，需討論。

原因：

- 這個方向和 FHIR 專案相似，但可能會讓專案範圍變大。
- 需要確認它是核心功能，還是進階亮點。
- 需要查詢是否已有類似資料標準，避免命名或定位不清。

待補資料：

- RESO Web API / RESO Data Dictionary 相關資料
- Schema.org RealEstateListing 相關資料
- 組員決定的 JSON schema 草案
- 是否作為 Demo 匯出功能

### 10. 地區租金行情比較是否納入 MVP

目前狀態：可能會做，需查證。

原因：

- 需要有可用且可信的地區租金資料。
- 不同地區、坪數、房型可能差異很大，簡單平均可能造成誤判。
- 若資料不足，可能只適合作為進階功能。

待補資料：

- 租金行情資料來源
- 行政區、坪數、房型欄位是否完整
- 中位數或平均數的計算方式
- 異常租金判斷標準
